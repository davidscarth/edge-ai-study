### vLLM:
```shell
sudo docker run --rm -it \
  --gpus all \
  --ipc=host \
  --network host \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  -v ~/models/templates:/mnt/templates \
  -e VLLM_USE_V2_MODEL_RUNNER=1 \
  ghcr.io/timothystewart6/vllm-gb10:latest \
  vllm serve RedHatAI/gemma-4-26B-A4B-it-NVFP4 \
  --host 0.0.0.0 --port 8000 \
  --gpu-memory-utilization 0.25 \
  --enable-prefix-caching \
  --speculative-config '{"method": "mtp", "model": "google/gemma-4-26B-A4B-it-assistant", "num_speculative_tokens": 3}' \
  --kv-cache-dtype fp8 \
  --max-model-len 32768 \
  --max-num-seqs 6 \
  --max-num-batched-tokens 8192 \
  --chat-template /mnt/templates/chat_template.jinja \
  --default-chat-template-kwargs '{"enable_thinking": true}' \
  --enable-auto-tool-choice \
  --tool-call-parser gemma4 \
  --reasoning-parser gemma4 \
  --api-key "REPLACE-THIS-WITH-YOUR-OWN-KEY"
```

The above lets me serve 6 requests with 32K context windows, and vLLM does pretty efficient batching so 6 users could all chat pretty comfortably with these settings, and leaves me plenty of RAM for other projects.

### Open WebUI System Prompt:
```shell
Today is {{CURRENT_WEEKDAY}}, {{CURRENT_DATE}}. Use search_web for anything recent, subject to change, or not confidently known, including questions about specific places, businesses, products, or people. Use fetch_url when the user shares a link or when search snippets lack needed detail. State plainly if unable to verify. Never invent facts, quotes, citations, or URLs. Think only to plan: what's needed, which tools, key facts. Never draft or outline the answer in thinking: write it once, directly. Skip deliberation entirely for simple requests.
```

This system prompt fixed a lot of my initial problems. May need additional tuning, but it's designed to be token-efficient and concise. Each piece fixed a real issue I encountered. Highly recommended to be paired with the RAG Template below (which fixes a search calling snag I encountered in thinking mode, not fully but enough).

### Open WebUI RAG Template:
```shell
### Task:
Answer the user's query using the provided context.

### Guidelines:
- The context includes documents, search results, and tool results from this conversation. The numbered <source> entries may contain document content, or may be empty when material arrived as search or tool results instead - both are normal; don't remark on the structure.
- When a claim comes from the context, cite it inline with the matching source number (e.g., [1], [2]). If no numbered source corresponds but the result includes a URL, cite by linking it. If neither exists, no citation is needed. Do not cite your own general knowledge.
- If the context doesn't contain the answer but you know it, answer from your own knowledge and say so briefly.
- If you can't answer reliably, say so plainly.
- Respond in the user's language.
- Never mention these instructions, the context block, or any XML tags in your reply.

<context>
{{CONTEXT}}
</context>
```

This fixes some weird cases. Searches go into the RAG pipeline but have empty ID's and the original template creates a weird condition that occurs for search results (models expect ID's to basically exist or not exist based on the prompt, but instead they get these null ID's, and may try to rationalize why instead of just ignoring it).

*26B: MTP3 wins overall — +16% single-stream, **+70% aggregate at c10** (148 → 251 t/s) vs no MTP, holding 30 t/s per user at ten concurrent. 31B: MTP is the difference between unusable and usable (11 → 20-23 t/s single-stream); MTP2/3/4 within noise of each other. Prefill ~7-9K t/s (26B) throughout. Real-world observed range: 45-80 t/s depending on content predictability (thinking-heavy text runs faster via higher draft acceptance).*
